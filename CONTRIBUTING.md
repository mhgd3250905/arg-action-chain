# 贡献指南

欢迎贡献新的 Step Contract 示例、反模式、验证命令设计和真实踩坑复盘。

## 适合贡献什么

- 新的 Step Contract 模板
- 某类任务如何拆 step 的案例
- 验证命令或 validator 设计
- 失败处理策略
- 与 ReAct、Workflow、DAG、多 Agent 等范式的对比
- 文档错别字、表达修正和结构优化

## 不建议贡献什么

- 需要私有账号、私有接口或私有数据才能理解的案例
- 没有验证机制的“Agent 提示词”
- 只展示成功路径、没有失败处理的流程
- 把 ARG 行动链路包装成万能 Agent 框架的宣传稿

## 写作原则

- 讲清楚任务边界，不要只堆概念。
- 每个 step 都要回答：输入、任务、输出、验证、失败处理、下一步。
- 能机器验证就写机器验证；不能机器验证就承认它需要软检查或人工确认。
- 避免泄露真实业务系统的账号、Token、邮件、IP、内部路径和客户数据。

## 推荐 PR 结构

```md
## 变更内容
- ...

## 为什么需要
- ...

## 如何验证
- ...

## 是否包含敏感信息
- 否 / 已脱敏
```
