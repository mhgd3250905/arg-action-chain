# ARG 行动链路 — 一页摘要

## 一句话
**把 Agent 变成一个 ARG 玩家：只看到当前线索卡，解谜通过验证后才能看下一张。Agent 是解谜者，不是出题人。**

## 解决什么问题
LLM Agent 在多步任务中会：超前规划、行动发散、自我欺骗。根源是**视野和自由度太大**。

## 三个核心约束

| 约束 | 机制 | 解决什么 |
|------|------|---------|
| **视野约束** | 逐步揭示：Agent 只读当前 step 文件，不知道下一步 | 超前规划、跳过步骤 |
| **行动约束** | 任务写成 bash 命令 + 枚举判断表 | 自由发挥、发明标签、幻觉 |
| **验收约束** | 可执行验证命令（exit code = 真相源） | Agent 声称完成但产出无效 |

## 线索卡六字段

```
输入 → 任务 → 输出 → 验证命令 → 验证失败处理 → 下一步
```

Agent 可以看到当前契约中的验证要求，但不能修改验证逻辑；验收结果由可执行命令、断言和 exit code 决定。

## 何时用 / 何时不用

| ✅ 适合 | ❌ 不适合 |
|---------|----------|
| 多步数据流水线 | 单步简单查询 |
| 有结构化产出 | 开放式研究/探索 |
| 需跨步一致性 | 多轮交互对话 |
| 定时自动执行 | 创意写作 |

## 与其他范式的关系

- vs. ReAct：去掉了推理环节，线索卡已规定好下一步
- vs. DAG Workflow：Agent 看不到全貌 = 不会想改流程
- vs. Plan-and-Execute：策划权交给人，Agent 只管执行
- vs. 多 Agent：ARG 约束单 Agent 边界，可组合使用

## 一句话原则

> 把 Agent 能看到的、能做的、能说的，压缩到一个 Step Contract 的范围内。它每通过一次验证，才获得看到下一步的资格。
