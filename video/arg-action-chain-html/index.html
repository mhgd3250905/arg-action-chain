ARG
行动链路

一套用 Step Contract 控制 Agent 发散、幻觉和自我欺骗的工程设计方案。

逐步揭示 契约执行 验证门禁 格子化判断
Problem

Agent 的问题不是不聪明

恰恰相反，很多问题来自它太想“帮忙”。

1
超前规划你让它做第三步，它已经开始重写第七步。
2
行动发散模糊指令会诱发新标签、新字段和额外动作。
3
自我验收自己产出，自己检查，然后说“看起来没问题”。
REQUEST请处理当前数据
AGENT我顺便优化了流程
OUTPUT多了三个你没要的字段
CHECK我觉得已经完成
Shift

别把 Agent 当架构师。
把它变成 ARG 玩家。

玩家只看到当前线索卡。解开、验证、通过，才获得下一张。

Step Contract

每张线索卡，必须回答六件事

输入从哪里拿数据，不让 Agent 猜。
任务只做当前一步，不顺手扩展。
输出产物是什么，路径是什么。
验证命令由脚本、断言或门禁验收。
失败处理重试、跳过、降级或终止。
下一步通过后解锁什么，或 TERMINAL。
Boxing

LLM 可以判断，但必须进格子

语义能力保留，自由发挥拿掉。

For each item, fill:
relevant: yes / no / uncertain
category: bug / feature / question / other
severity: high / medium / low
summary: 80 字以内中文

raw content, url, author: script-owned
Validation

验收权不在 Agent 手里

Agent 可以执行任务，但不能靠自我声明通关。

✓
JSON 能解析结构有效，不是“看起来像”。
✓
枚举值合法不允许发明新标签。
✓
数量一致total 必须等于 items 长度。
python -c "
assert d['total'] == len(d['items'])
assert item['category'] in allowed
assert item['summary'].strip()
print('PASS')
"

What It Really Changes

它不能消灭模型错误。
但能让错误更少发生、更早暴露、更容易拦截。
