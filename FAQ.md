# FAQ

## ARG 行动链路是不是 Agent Workflow？

它更窄，也更硬。普通 workflow 关心任务顺序，ARG 行动链路额外强调 Agent 的视野、行动和验收边界。

## 逐步揭示是不是安全机制？

不是。它首先是工程约束，不是权限隔离。若 Agent 有权限扫描全部文件，它理论上仍能看到完整流程。严肃生产环境需要配合沙箱、权限控制或独立执行器。

## 验证命令一定要 100% 判断业务正确吗？

不一定。可机器判断的部分用硬校验；语义质量、审美、策略取舍这类问题要用软检查或人工确认。不要把不可判定的问题伪装成硬门禁。

## 为什么不让 Agent 自己规划？

因为这套范式服务的是可重复、可验证、多步骤任务。这里需要的是稳定执行，不是临场发挥。需要探索时可以用 ReAct 或研究型 Agent，别把锤子当瑞士军刀。

## LLM 的语义能力怎么保留？

只保留在有限字段里。例如让它判断 `category`、`severity`、`summary`，并规定枚举、字数、判例和输出 JSON 结构。

## 脚本和 Agent 怎么分工？

脚本负责确定性操作：抓取、清洗、合并、入库、渲染、校验。Agent 负责脚本难以完成的语义判断。别让 Agent 拼关键 JSON，也别让它改原始数据。

## 一个 step 多大合适？

一个 step 应该只做一类事，并且能独立验证。若验证失败时你说不清是哪部分坏了，这个 step 就太大。

## 可以和多 Agent 结合吗？

可以。每个 Agent 可以跑自己的 ARG 链，上层再做编排。但每个 Agent 内部仍应受 Step Contract 约束。

## 开源示例为什么不用真实业务？

因为真实业务资料容易包含敏感路径、账号、客户信息、策略口径和生产经验细节。开源应该传播方法，不应该裸奔生产系统。
