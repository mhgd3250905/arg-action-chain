# ARG 行动链路

> 把 Agent 变成 ARG 玩家：只拿当前线索卡，完成当前任务，通过验证后解锁下一步。

ARG 行动链路是一种面向 LLM Agent 的受控执行工程范式。它借鉴 ARG 游戏的逐步解锁结构，把复杂任务拆成一组只读的 Step Contract。Agent 每次只执行当前步骤，不能自行重写流程，不能跳过验证，也不能靠一句“我检查过了”宣布完成。

这套范式的目标很朴素：**不要幻想 Agent 会自觉可靠，而是把它的视野、行动和验收权都压缩到可控边界内。**

在实际使用中，ARG 行动链路的价值主要体现在：显著限制 Agent 的行动发散、幻觉输出和自我欺骗式验收。它不承诺让模型变得“不会错”，而是通过工程边界让错误更少发生、更容易暴露、更容易被拦截。

## 核心观点

Agent 在多步任务里最常见的三个问题是：

- **超前规划**：看到全局后开始“优化”人设计好的流程。
- **行动发散**：模糊指令下自由发挥，发明标签、改字段、写散文。
- **自我验收**：自己产出、自己检查、自己宣布通过。

ARG 行动链路的回答是：

- **逐步揭示**：Agent 只读取当前 step，不预知后续步骤。
- **契约执行**：每个 step 都是输入、任务、输出、验证、失败处理、下一步的完整契约。
- **格子化判断**：需要 LLM 语义能力时，只允许填写有限字段和枚举值。
- **验证门禁**：可执行命令、脚本、schema、exit code 接管验收权。
- **数据防火墙**：关键数据合并、入库、模板组装由确定性脚本完成，Agent 只写被允许的字段。

## 一张线索卡

每个 Step Contract 至少包含六个字段：

```md
## 输入
当前步骤需要读取的文件、接口或数据来源。

## 任务
可执行命令，或被严格限定的语义判断任务。

## 输出
当前步骤必须产出的文件、对象或可见结果。

## 验证命令
可执行检查。通过与否由 exit code、断言或脚本输出决定。

## 验证失败处理
失败后是重试、降级、跳过，还是终止。

## 下一步
验证通过后要读取的下一个 step；或 TERMINAL。
```

Agent 的角色不是设计师、不是架构师、不是验收员。它只是当前线索卡的执行者。

## 什么时候适合

适合：

- 多步数据流水线
- 定时自动执行任务
- 有结构化产出的采集、清洗、分析、报告任务
- 需要 LLM 做少量语义判断，但又必须控制幻觉的场景
- 每步产物可以被脚本或规则验证的任务

不适合：

- 单次简单问答
- 开放式研究探索
- 高度交互式方案讨论
- 完全主观、无法验证的创作任务
- 需要 Agent 自主发现路径的复杂调查任务

别把它当万能框架。万能框架一般都很万能地失败。

## 最小实现

一个最小 ARG 行动链路通常长这样：

```text
SKILL.md / runner
  只保存循环引擎：
  读取当前 step -> 执行任务 -> 执行验证 -> 读取下一步 -> 直到 TERMINAL

plans/
  step-01-fetch.md
  step-02-process.md
  step-03-render.md

scripts/
  fetch.py
  merge.py
  validate.py
```

核心原则：**具体业务指令放在 step 文件里，确定性操作放在脚本里，Agent 只处理被限定的语义空格。**
