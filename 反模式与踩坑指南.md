# 反模式与踩坑指南

## 反模式 1：验证命令是自然语言

错误示例：

```md
请检查输出是否合理。
```

问题：Agent 会读一遍自己的产物，然后说“合理”。这不叫验证，这叫自我安慰。

更好的做法：

```bash
python -c "
import json
d=json.load(open('output/final.json'))
assert d['total'] == len(d['items'])
print('PASS')
"
```

## 反模式 2：把所有步骤写进 SKILL.md

如果引擎文件里写满所有业务步骤，Agent 一开始就看到了全局，视野约束直接报废。

更好的做法：

```text
SKILL.md
  只放循环引擎

plans/
  step-01-fetch.md
  step-02-classify.md
  step-03-render.md
```

## 反模式 3：一步做太多事

错误示例：

```text
Step 01: 拉取数据、清洗、分类、入库、生成报告、通知用户
```

更好的拆法：

```text
Step 01: 拉取
Step 02: 分类
Step 03: 入库
Step 04: 渲染
Step 05: 通知
```

## 反模式 4：验证太松

错误示例：

```bash
[ -s output.json ] && echo PASS
```

文件存在不代表内容正确。至少检查 JSON 结构、字段范围、数量一致性和关键字段非空。

## 反模式 5：Agent 可以修改原始数据

错误做法：让 Agent 在判断时顺便修正原文、作者、URL、时间。

正确做法：Agent 只写授权字段。原始字段由合并脚本从 raw 数据复制。

## 反模式 6：没有失败处理

错误示例：

```md
## 验证失败处理
重试。
```

重试几次？失败后跳过还是终止？是否需要人工确认？不写清楚，Agent 就会开始自由发挥。

## 反模式 7：下一步不明确

每条链都必须有明确的 `TERMINAL`。最后一步没有终点，Agent 很容易继续编故事。

## 反模式 8：把不可机器判断的问题硬塞进 validator

例如“报告是否足够有洞察力”。这种问题不该伪装成 `assert`，应该变成人工确认、评分表或软检查清单。

## 反模式 9：路径依赖当前目录

生产链路里尽量使用明确路径或统一变量。相对路径在不同执行环境下很容易翻车。

## 反模式 10：把 ARG 行动链路包装成万能框架

它不是万能框架。它适合可重复、可验证、结构化、多步骤任务。开放探索和创意任务硬套这个范式，只会把事情搞得又笨又慢。
